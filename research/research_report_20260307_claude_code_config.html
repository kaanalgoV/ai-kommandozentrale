Executive Summary

Claude Code laesst sich durch ein Zusammenspiel aus fuenf Schichten zu einer vollstaendigen AI-Agent-Kommandozentrale ausbauen: (1) CLAUDE.md als zentrale Konfiguration, (2) Skills fuer spezialisiertes Wissen, (3) Hooks fuer automatisierte Workflows, (4) Subagents und Agent Teams fuer parallele Arbeit, und (5) MCP-Server fuer externe Tool-Integration [1][2][3].

Der entscheidende Durchbruch im Februar 2026 war die Einfuehrung von Agent Teams, die mehrere Claude-Code-Sessions gleichzeitig an verschiedenen Aufgaben arbeiten lassen, wobei ein Team Lead die Koordination uebernimmt und Teammates direkt miteinander kommunizieren [4][5]. Durch Git Worktrees arbeiten diese parallelen Sessions isoliert in eigenen Branches [6][7].

1. CLAUDE.md: Das Fundament

1.1 Struktur und Platzierung

CLAUDE.md ist die wichtigste Konfigurationsdatei. Claude Code laedt automatisch alle CLAUDE.md-Dateien entlang der Verzeichnishierarchie -- von der Root bis zum aktuellen Arbeitsverzeichnis [1][11]. Fuer Monorepos: eine Root-CLAUDE.md mit globalen Regeln und subdirectory-spezifische CLAUDE.md-Dateien fuer paketspezifische Anweisungen.

Der empfohlene Einstieg ist /init, das eine Starter-CLAUDE.md basierend auf der Projektstruktur generiert [1]. Kritische Regel: Fuer jede Zeile fragen "Wuerde Claude Fehler machen, wenn diese Zeile fehlt?" -- wenn nein, streichen. Aufgeblaehte CLAUDE.md-Dateien fuehren dazu, dass Claude Anweisungen ignoriert [1][12].

1.2 Optimale Inhalte

Eine effektive CLAUDE.md enthaelt [1][11][12]:

  • Bash-Befehle: Build-, Test-, und Lint-Kommandos
  • Code-Style-Regeln: Namenskonventionen, Import-Reihenfolge, Error-Handling
  • Workflow-Regeln: "Tests vor Implementierung", "Nie direkt auf main pushen"
  • Projektkontext: Tech-Stack, Architekturentscheidungen

Fortgeschrittene Konfiguration nutzt .claude/rules/ fuer thematische Trennung (z.B. testing-rules.md, api-conventions.md) [11].

1.3 Routing-Framework

Ein Quick Win ist ein Routing-Framework direkt in der CLAUDE.md [13]: 3+ unabhaengige Tasks ohne Shared State gehen an parallele Subagents, Tasks mit Abhaengigkeiten werden sequentiell ausgefuehrt, Research geht an Background Agents, und komplexe Multi-File-Features an Agent Teams.

2. Skills: Spezialisiertes Wissen on-demand

2.1 Erkennungsmechanismus

Skills sind Ordner mit Instruktionen, Skripten und Ressourcen, die Claude dynamisch laedt [14][15]. Der entscheidende Punkt: Es gibt kein algorithmisches Routing. Claude Code formatiert alle Skill-Beschreibungen in einen Text und laesst das Sprachmodell entscheiden [15].

2.2 Speicherorte

ScopePfadGeltungsbereich
Personal~/.claude/skills/Alle Projekte, nur fuer dich
Projekt.claude/skills/Alle, die das Repo klonen

2.3 Zuverlaessige Erkennung

Damit Skills zuverlaessig erkannt werden [14][15]: Praezise description im YAML-Frontmatter, explizite Trigger-Woerter ("Use when user says 'deep research'"), negative Trigger ("Do NOT use for simple lookups"), und Live-Detection -- Skills koennen waehrend einer Session bearbeitet werden ohne Neustart [14].

3.2 Permissions-Allowlist

Allow/Deny-Listen reduzieren Bestaetigungsdialoge [18][19]. Claude geht die Liste von oben nach unten durch. Tool-Namen unterstuetzen Patterns wie Bash(git commit:*). Der Befehl /permissions ermoeglicht Echtzeit-Verwaltung [18].

Kosten-Tipp: CLAUDE_CODE_SUBAGENT_MODEL steuert das Subagent-Modell. Hauptsession auf Opus, Subagents auf Sonnet -- spart erheblich bei den Kosten [13].

4. Hooks: Automatisierte Workflows

Hooks sind Shell-Commands, HTTP-Endpoints oder LLM-Prompts an bestimmten Lifecycle-Punkten [20][21][22]. 15 Events und 3 Handler-Typen machen es zum umfassendsten Hook-System unter AI-Coding-Tools.

EventWannEinsatz
PreToolUseVor Tool-AusfuehrungValidierung, Blocking
PostToolUseNach Tool-AusfuehrungFormatting, Tests, Logging
UserPromptSubmitNach User-EingabeTask-Routing, Context
StopAgent fertigZusammenfassung, Notification
SessionStartSession beginntMemory laden, Context restore

Hooks sollten versioniert unter .claude/hooks/ liegen, valides JSON sein, idempotent arbeiten und als executable markiert sein [13][22].

5. Subagents: Spezialisierte Arbeiter

Subagents werden als Markdown mit YAML-Frontmatter in .claude/agents/ definiert [13][23]. Jeder laeuft in eigenem Context-Window mit eigenem System-Prompt und spezifischem Tool-Zugang.

Tool-Scoping Best Practices

Agent-TypTools
PM / ArchitectRead-heavy: Search, Docs via MCP
ImplementerEdit, Write, Bash + UI-Testing
ResearcherWebSearch, WebFetch, Read
ReleaseNur Deployment-relevante Tools

6. Agent Teams: Die Kommandozentrale

Agent Teams (Februar 2026) erlauben mehrere koordinierte Claude-Sessions [4][5][24]. Aktivierung via CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 in settings.json oder als Environment-Variable.

Architektur

  • Team Lead: Hauptsession -- erstellt Tasks, spawnt Teammates, synthetisiert Ergebnisse
  • Teammates: Unabhaengige Sessions mit eigenem Context-Window
  • Shared Task List: Dependency-Tracking mit Auto-Unblock
  • Mailbox: Direkte Kommunikation zwischen Teammates
Empfehlung: Split Panes (tmux/iTerm2) statt In-Process -- jeder Teammate bekommt ein eigenes Panel. Ab 2+ Teammates die einzige sinnvolle Option [4][5].

7. Multi-Projekt mit Git Worktrees

Git Worktrees erstellen separate Arbeitsverzeichnisse mit eigenen Dateien und Branches bei geteilter Repository-History [6][7][25].

claude --worktree feature-auth          # Eigener Worktree + Branch
claude --worktree feature-auth --tmux   # Mit tmux-Session

Beim Beenden: Keine Aenderungen = automatisches Cleanup. Aenderungen vorhanden = Prompt ob behalten [6]. Subagents koennen ebenfalls Worktree-Isolation nutzen [6].

10. Empfehlungen

Sofort umsetzen

  1. CLAUDE.md mit /init erstellen und auf unter 200 Zeilen kuerzen
  2. Permissions-Allowlist in ~/.claude/settings.json einrichten
  3. Deep-Research-Skill und Orchestra-Research-Skills installieren
  4. GitHub MCP-Server und Context7 anbinden

Naechste Schritte

  1. 3-5 projektspezifische Subagents in .claude/agents/ definieren
  2. PostToolUse-Hooks fuer Auto-Formatting einrichten
  3. Agent Teams aktivieren und mit 2-3 Teammates testen
  4. Git Worktrees fuer Multi-Feature-Arbeit etablieren

Limitierungen

  • Agent Teams sind experimentell (Stand Maerz 2026) und koennen sich aendern [4]
  • Skill-Erkennung ist nicht deterministisch -- Claude entscheidet per Sprachmodell [15]
  • Context-Limits bleiben trotz MCP Tool Search endlich [10]
  • Worktree-Cleanup kann bei Abbruechen fehlschlagen [6]
  • Fehlerhafte Hooks koennen den Workflow blockieren [22]
